รายการข้อมูลที่ ยังไม่มีในชุดข้อมูลปัจจุบัน แต่ถ้าได้มาจะปิดข้อจำกัดที่ระบุไว้ใน REPORT_Methodology_Verification.html และยกระดับความน่าเชื่อถือของรายงานจาก "ประมาณการที่สมเหตุสมผล" ไปเป็น "ยืนยันได้ด้วยข้อมูลจริง"
Prepared by กทท. Planning Division · จัดทำ 23 มิถุนายน 2569 · ต่อเนื่องจาก REPORT_Methodology_Verification.html และ METHODOLOGY_DwellTime.md (Part B research agenda)
สรุปสำหรับผู้บริหาร

แบ่งความต้องการข้อมูลเป็น 3 ระดับตามความยากในการได้มา ไม่ใช่ตามความสำคัญ — ของบางตัวได้ง่าย (อยู่ในมือกองอื่นในองค์กรแล้ว) แต่ของที่ ROI สูงสุด ต้องขอข้ามหน่วยงาน เพราะปิดจุดอ่อนที่สุดของรายงานทั้งชุดตอนนี้คือ "ตัวเลขรายได้เป็นค่าประมาณ ไม่ใช่ของจริง"

ไม่มีรายการใดจำเป็นต้องทำทันที — งานปัจจุบันใช้งานได้และตรวจสอบย้อนกลับได้ครบตามที่มี เอกสารนี้คือ wishlist สำหรับรอบทำงานถัดไป เผื่อมีโอกาสขอข้อมูลเพิ่ม

1ได้ง่าย — น่าจะมีอยู่ในมือกองอื่นแล้ว
ไม่ต้องขอข้ามองค์กร แค่ขอข้ามแผนก/กอง
ทะเบียนรถ / ผู้ขนส่ง (Transporter ID)ได้ง่าย
ใช้ปิดข้อจำกัด #5 ใน Methodology Report — Section 6 "ตู้วน" ปัจจุบันพิสูจน์ได้แค่ระดับเลขตู้ ไม่ใช่ระดับรถ/ผู้ประกอบการ
ยกระดับเป็นจัดอันดับได้ว่าผู้ประกอบการรายไหน "บริหารตู้เปล่าเก่งสุด" จริง ไม่ใช่อนุมานจาก pattern เฉลี่ยทั้งระบบ
แหล่งที่เป็นไปได้ระบบ gate scan ของ ทกท. เอง (มักมีทะเบียนรถ/บริษัทขนส่งติด log การเข้า-ออกอยู่แล้ว)
Customs Clearance Timestampได้ง่าย
ใช้ปิดคำถามที่ Section 1-2 ตอบไม่ได้ตอนนี้ — "ตู้ค้างนานเพราะรอพิธีการ" หรือ "พิธีการเสร็จแล้วแต่ลูกค้าไม่มารับ" คนละสาเหตุ คนละทางแก้
ยกระดับเป็นแยก dwell time เป็น 2 ช่วงย่อย (รอพิธีการ vs รอลูกค้า) ทำให้ recommendation เจาะจงขึ้นมาก
แหล่งที่เป็นไปได้ระบบศุลกากร (กรมศุลกากร) หรือ TOS เองถ้ามี flag สถานะอยู่แล้ว
2ต้องขอข้ามหน่วยงาน — แต่คุ้มมาก
ใช้เวลาประสานงานนานขึ้น แต่ปิดจุดอ่อนใหญ่สุดของรายงาน
Revenue / GL จริงระดับเดือนต่อตู้ROI สูงสุด
ใช้ปิดข้อจำกัด #1 ใน Methodology Report — ตัวเลขรายได้ทั้งชุด (+99.3 ล้าน FY68, +115 ล้าน FY69) เป็นค่าประมาณจากอัตราค่าภาระตามหนังสือ พ.ศ.2543 ไม่ใช่รายได้จริงที่บันทึกบัญชี
ยกระดับเป็นเปลี่ยนรายงานจาก "น่าเชื่อ" เป็น "ยืนยันได้" — ถ้าตัวเลขจริงตรงกับค่าประมาณ ยืนยัน methodology ทั้งหมด ถ้าไม่ตรง จะเจอช่องว่างที่ต้องอธิบาย (เช่น discount/ส่วนลดที่ไม่ได้อยู่ในหนังสืออัตรา)
แหล่งที่เป็นไปได้ฝ่ายการเงิน/บัญชี — Power BI GL export ที่ใช้อยู่แล้วใน Financial Dashboard repo/BKP_Workspace.html อาจมีรายละเอียดระดับนี้อยู่แล้ว แค่ยังไม่ได้ join กับข้อมูล dwell
Vessel Schedule (ตารางเรือ)ปานกลาง
ใช้ปิดข้อจำกัด #4 — ข้อมูลชั่วโมง/วันฝั่ง Export ไม่น่าเชื่อถือตอนนี้เพราะ 99.8% เป็น batch event ที่ถูกกรองออก
ยกระดับเป็นผูก Export dwell กับรอบเรือจริง แทนการอนุมานจาก batch pattern — ตอบได้ว่า "ตู้ค้างเพราะรอเรือเที่ยวต่อไป" จริงหรือไม่ ไม่ใช่แค่สมมติฐาน
แหล่งที่เป็นไปได้ฝ่ายปฏิบัติการเรือ/สายเดินเรือ
3เชิงคุณภาพ — ไม่ใช่ข้อมูลตัวเลข
ต้องอาศัยคนยืนยัน ไม่สามารถดึงจากระบบได้
ยืนยัน Batch Event ที่ยังค้างคาใจเชิงคุณภาพ
สถานะปัจจุบันเจอ cluster ตู้จำนวนมากที่ time_out ตรงกันเป๊ะถึงวินาที 2 จุดใหญ่ใน Anomaly tier (>30 วัน) — เช่น 2026-01-31 08:05 (438 ตู้), 2023-09-30 16:10 (371 ตู้)
ทำไมต้องยืนยันยังสรุปไม่ได้ว่าเป็น system close-out artifact (ไม่ใช่ dwell จริง) หรือ เรือเที่ยวใหญ่ที่ขนตู้พร้อมกันจริง — สองกรณีนี้ตีความ Anomaly คนละแบบ
ใครยืนยันได้ทีม yard operation ที่ดูแลระบบ gate scan / TOS หน้างาน — ไม่กระทบ KPI หลักตอนนี้ (อยู่นอก Normal segment อยู่แล้ว) แต่กระทบความแม่นยำของ Anomaly tier โดยตรง
★คำแนะนำ — ถ้าเลือกทำได้แค่ 1 อย่างก่อน
ROI สูงสุด
Revenue / GL จริงระดับเดือนต่อตู้
เพราะตัวเลขรายได้เป็นหัวใจของ Infoboardgraphic ทั้งฉบับ (ตอบคำถามตั้งต้น "ทำไมรายได้เพิ่มขึ้น") แต่ปัจจุบันคำนวณจากอัตราค่าภาระตามหนังสือ พ.ศ.2543 เท่านั้น — มีโอกาสสูงที่ฝ่ายการเงินมีข้อมูล GL ระดับนี้อยู่แล้วในมือ (เห็นได้จาก Financial Dashboard repo ที่ใช้ Power BI GL export อยู่แล้ว) จึงเป็นรายการที่ ขอได้ง่ายที่สุดในกลุ่ม ROI สูง และเปลี่ยนสถานะรายงานจาก "ประมาณการที่สมเหตุสมผล" เป็น "ยืนยันได้ด้วยตัวเลขจริง" ได้ทันทีที่ได้ข้อมูล
ลำดับรายการระดับปิดข้อจำกัด
1Revenue/GL จริงต่อตู้ต้องขอข้ามหน่วยงาน#1 (รายได้เป็นค่าประมาณ)
2ทะเบียนรถ/ผู้ขนส่งได้ง่าย#5 (พิสูจน์ได้แค่ระดับเลขตู้)
3Customs clearance timestampได้ง่ายแยกสาเหตุ dwell (รอพิธีการ vs รอลูกค้า)
